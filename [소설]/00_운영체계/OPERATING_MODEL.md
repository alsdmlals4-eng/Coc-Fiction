# Coc-Fiction 소설 운영 모델

## 1. 목적

긴 소설 작업을 여러 채팅과 도구가 이어 받아도 다음을 잃지 않게 한다.

```text
사용자 의도
→ 작품 코어·정본
→ 플롯·장면 설계
→ 장면 집필
→ 구조·연속성·문장 퇴고
→ 교정·검증
→ 현재 상태·다음 작업·학습
```

## 2. 작품 책임 계층

```text
최신 사용자 지시
→ 승인된 작품 코어
→ 설정·인물·연표·플롯·문체 정본
→ 승인된 장면 목록·장면 카드
→ 실제 원고
→ 연구·참고자료·독자 피드백
```

한 질문에는 현행 책임 원본 하나만 둔다. 같은 사실을 여러 파일에서 독립적으로 관리하지 않는다. 충돌 시 최신 사용자 승인과 Canon 상태를 확인하고 임의로 평균내지 않는다.

## 3. 작품 코어 계약

작품 코어는 바꾸면 같은 작품으로 보기 어려운 최소 정체성이다.

- 독자에게 주는 핵심 약속
- 중심 인물과 가장 중요한 욕망·결핍
- 중심 갈등과 이야기 질문
- 주제 질문과 작품이 피할 단순한 정답
- 장르·정서·톤의 불변 범위
- 기본 POV·시제·서술 거리
- 결말의 방향 또는 금지된 결말
- 세계 규칙과 인물성의 변경 금지 항목
- 자유롭게 바꿀 수 있는 외피

상태는 `CORE_SEED / CORE_PROPOSED / CORE_STRESS_TESTED / CORE_CONFIRMED / CORE_REOPENED`를 사용한다. `CORE_CONFIRMED`는 사용자 승인 없이는 부여하지 않는다.

## 4. 작업 생명주기

```text
동시 작업 상태 확인
→ Prompt 의도·Manuscript Stage 파악
→ PLAN / BUILD / REVIEW 선택
→ Registry trigger로 최소 Skill·mode 선택
→ 범위·제외·보호·완료·검증 계약
→ 필요 시 결과 단위·의존성·순서 분해
→ 실제 작업
→ 적대적 검토·정합성·회귀 검증
→ 정본·현재 상태·체크포인트 갱신
→ PR 전 최신 main·겹치는 파일 재확인
```

## 5. 설계에서 원고까지

```text
작품 코어
→ 한 문장 전제
→ 인물 욕망·장애·대가·변화
→ 인과 플롯과 정보 공개 순서
→ 장면 목록
→ 장면 카드
→ 초고
→ 역개요
→ 다층 퇴고
→ 최종 교정
```

장편의 복잡도가 높을 때만 점진 확장형 설계를 사용한다. 한 문장→단락→인물·플롯→장면 목록으로 확장하는 Snowflake 방식은 선택 도구이며 강제 공식이 아니다.

## 6. 장면 카드 최소 계약

- 장면 ID·위치·시간
- POV 인물·서술 거리
- 장면 시작 상태
- 인물이 지금 원하는 것
- 장애·반대 힘·대가
- 핵심 전환 또는 발견
- 장면 종료 상태와 다음 행동 압력
- 새로 공개한 정보와 숨긴 정보
- 인물·관계·소지품·부상·지식·세계 상태 변경
- 반드시 보존할 이미지·대사·정서
- 삭제 또는 병합 가능한 이유

모든 장면이 동일한 공식일 필요는 없지만, 장면 전후에 무엇이 달라졌는지는 설명할 수 있어야 한다.

## 7. 퇴고 순서

작은 문장부터 고치지 않는다. 큰 층이 안정된 뒤 다음 층으로 내려간다.

1. **Developmental**: 작품 약속, 주제, 인물 아크, 결말, 전체 플롯
2. **Structural·Causal**: 장면 순서, 인과, 정보 공개, 긴장, 서브플롯
3. **Continuity·Scene**: 시간·공간·지식·행동 동기·POV·장면 목적
4. **Line·Style**: 명료성, 리듬, 반복, 추상어, 시점 거리, 대화·묘사
5. **Copyedit**: 맞춤법, 띄어쓰기, 문장 부호, 표기, 고유명사
6. **Proofread**: 최종본의 누락·오탈자·레이아웃

각 패스는 한 번에 2~3개 핵심 의제만 다룬다. 구조가 바뀌면 하위 패스의 결과를 다시 검토한다.

### 장편 묶음 패스

5화 등 연재 묶음을 원본·Canon과 대조해 수정할 때는 `fiction-revision-and-validation: serial-arc-pass`를 사용한다. 직전·직후 경계 화와 원본 범위를 작업 계약에 고정하고, Scene Card·원고·색인·역개요 override·Scene Pass Registry·Revision Report·활성 기획 문서를 같은 변경 범위에서 갱신한다. 자동 분석은 finding 후보로만 사용하며, 완료 묶음에서 발견한 규칙은 동일 결함이 확인된 범위에만 적용한다.

## 8. 적대적 검토·개선 루프

```text
attack
→ validate-critique
→ MUST_FIX / SHOULD_FIX / DEFER / REJECT / UNVERIFIED
→ 승인된 문제만 최소 수정
→ regression-recheck
→ decision-report
```

공격 관점은 모순, 인과 단절, 편의적 우연, 인물 붕괴, 설명 과잉, 정보 은폐의 부정직함, 감정 비약, 장르 약속 위반, 진부함, 문체 균질화, 독자 혼란을 포함한다.

비판도 취향·잘못된 전제·범위 밖 요구일 수 있다. 검증되지 않은 지적을 자동 반영하지 않는다. 수정 뒤 기존 장점·복선·인물성·톤이 훼손되지 않았는지 다시 공격한다.

## 9. 자료·참고문장 사용

- 공식·1차 자료를 우선하고 날짜·버전·범위를 기록한다.
- 참고문장은 짧은 인용 또는 사용자가 제공한 텍스트만 보관한다.
- 문장 자체를 복제하지 않고 `기능 → 장치 → 효과 → 전이 가능한 원리`로 분석한다.
- 살아 있는 작가의 고유 문체를 모사하는 지침을 만들지 않는다.
- 사실 자료, 작법 이론, 독자 반응, 작품 참고를 서로 다른 증거 유형으로 구분한다.
- 외부 자료가 작품 설정과 충돌해도 자동으로 Canon을 바꾸지 않는다.

## 10. 동시 작업과 Git

- `main`은 통합 기준이며 작업은 고유 브랜치에서 한다.
- 시작·PR 직전에 최신 main, 열린 PR, 겹치는 파일을 확인한다.
- 원고 이전과 운영체계 설치는 경로와 브랜치를 분리한다.
- `DIRTY / REMOTE_AHEAD / LOCAL_AHEAD / DIVERGED / BLOCKED`를 구분한다.
- force push·hard reset·무검토 덮어쓰기를 금지한다.
- 겹치는 변경은 `KEEP / MERGE / UPDATE / DEFER`로 판정하고 근거를 남긴다.

## 11. 완료 상태

- `PASS`: 요구된 검수를 실제 수행했고 차단 문제가 없다.
- `PARTIAL`: 일부 완료 또는 비차단 미검증이 있다.
- `FAIL`: 차단 결함이나 계약 위반이 있다.
- `NOT_RUN`: 환경·입력·권한이 없어 실행하지 않았다.
- `UNVERIFIED`: 근거가 부족해 판정할 수 없다.

문서가 존재한다는 사실과 실제 원고 품질·정합성·독자 반응이 검증됐다는 주장을 구분한다.
